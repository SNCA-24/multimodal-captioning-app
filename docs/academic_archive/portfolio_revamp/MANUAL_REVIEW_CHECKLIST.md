# MANUAL_REVIEW_CHECKLIST

## 1. 60-second skim test
Can a recruiter quickly understand:
- what the project is
- what was built
- why it matters
- whether the repo looks serious

## 2. 3-minute deep-dive test
Can a technical reviewer see:
- clear repo structure
- runnable path
- results/evaluation
- tradeoffs
- engineering judgment

## 3. Fresh-clone test
From a clean machine or fresh environment:
- install works
- main run command works
- test or smoke test works

## 4. Root cleanliness test
Bad signs:
- class reports in root
- committed .env
- placeholder docs
- random output clutter
- broken paths
- notebook-only supported workflow
- obvious local machine assumptions

## 5. README quality test
README should:
- explain the project fast
- surface strongest technical signals
- include architecture/workflow
- include setup and run commands
- include results/metrics/assets
- include limitations/tradeoffs

## 6. Engineering proof test
Repo should ideally show:
- tests or smoke tests
- basic CI
- environment/dependency management
- clean configs
- clear results/assets

## 7. Portfolio-fit test
Does this repo improve the overall GitHub profile for:
- MLE first
- DE second
- DS third

## 8. Pin-worthiness test
Pin only if:
- you would be comfortable with a hiring manager opening it today
- README is strong
- root is clean
- run path works
- tests/CI exist at least minimally
- results are visible
- repo does not feel embarrassing on hygiene

## 9. Final manual decision
Choose one:
- Pin now
- Polish more, then pin
- Keep public but not pinned
- Low priority
